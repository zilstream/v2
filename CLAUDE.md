# Repository Guidelines

## Project Structure & Module Organization
The app is built on TanStack Start (Vite + TanStack Router) and deploys to Cloudflare Workers. File-based routes live in `src/routes` (e.g. `src/routes/__root.tsx`, `src/routes/tokens.$address.tsx`). The router bootstrap is `src/router.tsx`; the Vite config wires plugins in `vite.config.ts`. Global styles are in `src/styles/app.css` and imported via `src/routes/__root.tsx` (`@tailwindcss/vite` handles Tailwind v4). ShadCN UI primitives stay in `src/components/ui`; project-specific wrappers live beside their consumers. Shared assets belong in `public`. Lint/format rules are centralized in `biome.json`; TypeScript paths and strict mode live in `tsconfig.json`. `src/routeTree.gen.ts` is auto-generated by the TanStack Router plugin and should not be edited or linted.

## Build, Test, and Development Commands
Install dependencies with `bun install` to stay aligned with `bun.lock`. Start the dev server via `bun run dev` (Vite on port 3000). Produce an optimized build with `bun run build` (Vite build plus `tsc --noEmit`). Preview the built output locally with `bun run preview`. Deploy to Cloudflare Workers with `bun run deploy` (requires a configured `wrangler.jsonc`). Regenerate Cloudflare binding types with `bun run cf-typegen`. Run `bun run lint` before submitting changes; apply `bun run format` if formatting drifts.

## Coding Style & Naming Conventions
Code is TypeScript-first with 2-space indentation enforced by Biome. Use PascalCase for React components (`HomePage`), camelCase for variables, hooks, and helper functions, and UPPER_SNAKE_CASE only for exported constants. Co-locate feature-specific modules beneath their route or component folder and prefer the `@/` path alias to avoid brittle relative imports. Tailwind v4 utilities are available—order classes from layout → spacing → color to aid scanning—and prefer ShadCN component slots over ad-hoc DOM when an existing primitive fits. Use TanStack Router's typed `Link` (`to` + `params`) for internal navigation; reserve `<a>` for external URLs.

## Testing Guidelines
Automated tests are not configured yet. When introducing a test harness (e.g., Vitest or Playwright), document the new scripts and wire them through `bun run`. Until then, include manual verification steps in every PR and avoid merging without checking the home route render, navigation behavior, and asset loading. Add smoke tests alongside any new critical feature once tooling is in place.

## Commit & Pull Request Guidelines
Write concise, imperative commit subjects similar to the existing history (e.g., `Add staking rewards table`), and reference related issues in the body using `#123`. Keep changes focused per commit. Pull requests should summarize intent, list QA steps or commands executed, and attach screenshots or recordings for UI updates. Ensure linting and builds pass locally before requesting review.
